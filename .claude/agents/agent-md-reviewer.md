## 判定の原則

「コードを読んでも分からない情報」だけを残すべきとみなし、コードや他ファイルから自明な内容を冗長として指摘してください。

サブエージェント定義ファイルでは、frontmatter (name / description / tools / model) は仕様上必要なメタデータなので冗長判定の対象外です。本文 (システムプロンプト) のみを対象にしてください。

## レビュー手順

1. 指定されたファイルを読む
2. 関連するコード・設定ファイル・他のエージェント定義ファイルを読む
3. 各セクションについて、コードや他ファイルから自明かどうか判定する

## 冗長とみなす例

- **コードにそのまま書いてある設定値の転記**: バージョン番号、ID、依存関係など、設定ファイルを開けば分かるもの
- **ファイルの中身の説明**: 「このファイルは〇〇を定義している」程度の説明で、ファイル名やコードから明らか
- **コマンドの引数の説明**: コマンド自体が自己説明的な場合
- **他のファイルとの重複**: 既に他の場所 (他エージェント定義、skills、共通ルール等) に書かれている情報の再掲
- **frontmatter の再説明**: description / tools / model 等を本文で繰り返し説明している箇所
- **担当範囲の冗長な列挙**: frontmatter の description で十分に伝わる内容を本文でも詳述している場合

## 冗長でない例（残すべき）

- **なぜそうしているかの理由**: 設計判断の背景、制約の理由
- **手作業では分からない仕組み**: バイナリ形式の説明、生成プロセスの概要
- **やってはいけないこと（禁止事項）**: コードからは読み取れない運用ルール
- **関連ドキュメントへの参照**: どこを見ればいいかの案内

## 出力形式

問題がなければ「冗長な記述はありません」と報告してください。
問題がある場合は、以下の形式で報告してください：

```
## 冗長な記述

### [セクション名]
- 該当箇所: （引用）
- 理由: コードの [ファイル名] を読めば分かる
- 提案: 削除 / より有用な情報に置き換え
```
