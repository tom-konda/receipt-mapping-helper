# CSS 実装ルール

優先度が高い順に以下の通りとする。

1. CSS のルール設置順
2. CSS の書き方

## CSS のルール設置順

1. 共通ルール
2. カラースキームごとのルール（`@media (prefers-color-scheme: ...)` 内）

この順序は `app.css` と Svelte コンポーネントの scoped style の両方に適用する。

## CSS の書き方

Nested CSS を使用すること。

ネストは目安として 2〜3 階層まで。セレクタの特異性 (specificity) を必要以上に上げないため、また grep でクラス名から実体に到達できる可読性を保つため。それ以上深くなる場合はクラスを切り出すなど、構造の見直しを検討すること。

## ソースコードへのコメント

セレクタとプロパティから読み取れる WHAT は書かない。コードからは読み取れない WHY (背景・意図・制約) のみ残す。例:

- ブラウザ互換のための上書きや既知の不具合の回避
- なぜその値・仕様にしたかの判断理由
- 一見不自然に見えるが意図的な指定であることの補足

## 定義済み視覚部品

`app.css` に定義された CSS クラスで表現する。生の `<button>` に独自スタイルを当ててボタンを作らないこと。

### ボタン

ボタンの見た目にしたい要素には必ず `.btn` を付与する (基底クラス)。種類で区別したい場合は variant クラスを併用する (例: `class="btn btn-danger"`)。

基底クラス (ボタンの見た目にする全ての要素に必須):

| クラス | 効果 |
|---|---|
| `.btn` | ボタンの共通スタイル (角丸、border、padding、font 等)。`<button>` だけでなく `<a>` 等にも付けられる |

variant (任意、いずれか 1 つ。何も付けない場合は通常ボタン = `class="btn"`):

| 用途 | クラス | 使いどころ |
|---|---|---|
| 最優先 | `.btn-primary` | 画面の主アクション (**1 画面 1 個まで**。トップページとモーダル/ダイアログ内は例外) |
| 優先 | `.btn-secondary` | 補助的な前進アクション (次へ、閉じる等)。モーダル/ダイアログ内のボタンはここに分類する |
| 通常 | (variant なし) | ナビゲーション (前へ、戻る、カルーセル送り等) |
| 危険 | `.btn-danger` | 破壊的操作 (削除等)。確認ダイアログと組み合わせること |

修飾子 (トップページ専用、`.btn` + variant と併用):

| クラス | 効果 |
|---|---|
| `.btn-top` | 横幅いっぱい・大型化。トップページのメニュー項目に付ける (例: `class="btn btn-primary btn-top"`) |

新しい variant を追加したくなった場合は、まず既存で表現できないか検討し、必要なら `app.css` 側でクラスを増やす。各コンポーネントの scoped style で独自にボタン装飾を上書きするのは禁止。

ただし `:disabled` 擬似クラスのスタイル上書きは例外として許容する。理由:

- 未入力のままステップを進めさせるなど、ユーザーが誤操作する方が問題が大きいため、コンポーネント固有の文脈に応じて非活性表現を強調できるようにする
- `:disabled` 擬似クラスはフォーム要素にしか存在しないため、影響範囲が限定される

例外を使う際は、scoped style 内で `:disabled` のスタイルのみを上書きすること (通常時の装飾も同時に上書きしない)。

ファイル選択ボタン (`<input type="file">`) は構造上 `<button>` ではないが、`app.css` 側で `input` 要素に付いた `.btn-primary` を `::file-selector-button` 疑似要素に橋渡しするルールを定義済み。マークアップでは他のボタンと同様に `class="btn btn-primary"` を付与する。
